PROMPT:
--------

New Bauhaus editorial design style. Use a cream/off-white (#F5F0E8) background
with bold black typography and vivid signal-red (#E8231A) graphic elements.
The layout should feel structured with a visible light-grid of thin red or
grey lines in the background (like graph paper or a technical schematic grid).

  FOR RESOURCE-CONSTRAINED EDGE HARDWARE

--- MAIN GRAPHIC ELEMENT ---

A large abstract "T" or "TC" letterform rendered in solid signal-red
(#E8231A), occupying the central-right area of the poster, at roughly 50–60%
of the poster's height. The letterform should be styled like a bold logo mark —
geometric, flat, with hard edges — partially bleed off the right edge of the
poster.

Overlapping / interacting with the letterform, place a stylized circuit-path
or data-flow diagram drawn in thick red lines (serpentine/angular S-shaped or
Z-shaped path), representing data moving through a cryptographic pipeline.
The diagram should pass through or behind the large letterform, similar to the
snake-like path element in the reference image.

In the center intersection area of the circuit diagram, place a small schematic
symbol — either a lock icon, a key icon, or a minimal 3-node lattice/graph
(like a small XOR gate or hash function diagram) drawn in clean black outlines
with small red circular node markers.

right alignment), each in compact bold black sans-serif:

- Grid of thin horizontal and vertical lines should span the full poster as a

- The three feature labels should sit on a thin horizontal red rule/line.
- The overall feel: confident, technical, minimal — like a high-end HPC

- Aspect ratio: 1:1 (square), suitable for social media and print.

PROJECT DATA REFERENCE (for fact-checking poster content)
================================================================================

Project   : TinyCrypto
Type      : Zero-dependency, bare-metal C++17 cryptographic engine
Targets   : ARM Cortex-M0+ (RP2040), ARM Cortex-M4 (STM32F4xx), Xtensa LX6 (ESP32)
Build     : CMake cross-compilation pipeline with separate toolchain files
Algorithms: AES-256-CBC, ChaCha20, SHA-256, Curve25519 (Montgomery ladder)

Key Design Invariants:
  - No OS, no RTOS, no dynamic heap allocation (malloc/new forbidden)
  - No STL containers, no exceptions, no RTTI
  - Compiled with -nostdlib -nodefaultlibs -fno-rtti -fno-exceptions
  - All cryptographic state lives on the stack (compile-time bounded frames)
  - Explicit key zeroing via explicit_bzero() on every context destruction
  - Constant-time execution throughout (no secret-dependent branches)

Performance (x86-64 Software Baseline):
  SHA-256       : 164 MB/s
  ChaCha20 AVX2 : 1.11 GB/s  (4.3× speedup over scalar)
  Curve25519    : ~3,200 ops/sec DH exchange
  AES-256       : ~57 MB/s (ECB)
  Const-time overhead: <9%

Footprint Comparison vs Alternatives:
  OpenSSL   : >500 KB flash, >2000 KB SRAM — requires full OS
  mbedTLS   : ~50 KB flash, ~50 KB SRAM — partial RTOS
  wolfSSL   : ~20 KB flash, ~20 KB SRAM — partial RTOS
  TinyCrypto: <10 KB flash,   <4 KB SRAM — bare metal, no OS

Validation: All NIST CAVP / RFC test vectors PASS
  (FIPS 180-4 SHA-256, RFC 8439 ChaCha20, RFC 7748 Curve25519, FIPS 197 AES)

Pre-silicon validation: QEMU hardware emulation (ARMv6-M / ARMv7-M)
Physical target tested: Raspberry Pi Pico (RP2040, 133 MHz, 264 KB SRAM)
