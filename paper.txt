ABSTRACT

The accelerating deployment of Internet of Things (IoT) nodes, wireless sensor
networks, and autonomous edge computing systems has created an exigent demand
for lightweight, deterministic cryptographic primitives capable of operating
within severely constrained hardware environments. Conventional cryptographic
libraries, most notably OpenSSL, presuppose the existence of a fully featured
host Operating System (OS) capable of managing virtual memory, dynamic heap
allocation, and hardware abstraction layers — architectural requirements
entirely incompatible with microcontrollers possessing as little as 2 KB of
SRAM and a single-core processor operating at sub-100 MHz. This paper presents
TinyCrypto, a zero-dependency, bare-metal C++ cryptographic engine architected
explicitly for deployment on raw silicon. By rigorously eschewing all OS-level
abstractions, dynamic memory allocation (malloc/new), and external linkage
dependencies, TinyCrypto delivers high-speed, constant-time implementations of
AES-256-CBC, ChaCha20, SHA-256, and Curve25519 directly on physical hardware,
including the ARM Cortex-M0+ (Raspberry Pi RP2040), Xtensa LX6 (ESP32), and
ARM Cortex-M4 (STM32F4xx) instruction set architectures. A cross-compilation
pipeline orchestrated via CMake and hardware emulation using QEMU further
ensures portability and pre-silicon validation. We detail the system
architecture, design invariants, implementation strategy, and a rigorous
tripartite benchmarking methodology to empirically quantify execution latency
in clock cycles, SRAM consumption in bytes, and code-size footprint in kilobytes
across software, emulated, and physical hardware execution contexts.

Index Terms — Bare-metal firmware, ChaCha20, cross-compilation, Curve25519,
AES-256, embedded cryptography, heap fragmentation, IoT security, QEMU, SRAM,
zero-dependency.

I. INTRODUCTION
================================================================================

The global proliferation of Internet of Things (IoT) systems has escalated at
a compound annual growth rate (CAGR) exceeding 16%, with credible industry
forecasts projecting in excess of 75 billion connected devices by 2030 [1].
These systems pervade critical infrastructure: industrial control systems
(ICS), implantable medical devices, autonomous vehicle sensor arrays, smart
grid edge nodes, and environmental monitoring networks. The security posture
of this vast and expanding ecosystem, however, remains critically underdeveloped.
A substantial fraction of deployed IoT endpoints execute without transport-layer
encryption, authenticated key exchange, or cryptographic attestation, creating
systemic vulnerabilities exploitable by adversaries for data exfiltration,
command injection, and large-scale denial-of-service (DoS) amplification via
botnet aggregation [2].

The root cause is not ignorance of established cryptographic protocols but
rather a fundamental impedance mismatch between the computational demands of
those protocols and the hardware realities of resource-constrained endpoints.
A typical IoT sensing node may be built around a microcontroller such as the
Raspberry Pi RP2040, which offers 264 KB of on-chip SRAM and a dual-core ARM
Cortex-M0+ processor clocked at 133 MHz, or an ESP32, providing 520 KB of SRAM
with a dual-core Xtensa LX6 at 240 MHz. While these figures appear modest,
they are categorically incompatible with the runtime requirements of standard
cryptographic libraries. OpenSSL, for instance, requires a minimum of several
megabytes of heap-managed memory for SSL context initialization alone, in
addition to POSIX operating system interfaces (sockets, file I/O, threading
primitives) that are fundamentally absent from bare-metal environments [3].

Existing embedded-focused libraries, such as mbedTLS and wolfSSL, reduce the
memory footprint and eliminate POSIX dependencies; however, they commonly
operate best when layered atop a Real-Time Operating System (RTOS) such as
FreeRTOS or Zephyr, which introduces additional layers of memory management,
stack partitioning, and task scheduling overhead. This indirection consumes
scarce SRAM resources and introduces non-deterministic scheduling jitter,
which is unacceptable in hard real-time systems where cryptographic operations
must complete within bounded, predictable time windows [4].

Further complicating deployment on constrained hardware is the hazard of heap
fragmentation. Microcontrollers operating over extended periods — days, months,
or years in field-deployed configurations — are highly susceptible to memory
fragmentation when dynamic allocation primitives (malloc, calloc, new) are
used by cryptographic state machines. Without a sophisticated memory compactor
or garbage collector — both precluded by the bare-metal context — the allocator
state gradually degrades until a physically contiguous block of sufficient size
can no longer be found, resulting in fatal allocation failures and device
lockup [5].

This paper introduces TinyCrypto, a zero-dependency, statically-allocated C++17
cryptographic engine that operates directly on bare silicon without an
underlying OS or RTOS, and without any recourse to dynamic heap allocation.
TinyCrypto provides a complete symmetric and asymmetric cryptographic primitive
suite — including AES-256-CBC, ChaCha20-Poly1305, SHA-256, and an elliptic
curve foundation for Curve25519 — compiled via a CMake-driven cross-compilation
toolchain targeting multiple embedded instruction set architectures (ISAs). The
system is designed for pre-silicon validation via QEMU hardware emulation and
direct firmware flashing onto physical microcontroller targets.

The primary contributions of this paper are as follows:

  (1) A formal description of the zero-dependency architectural design philosophy
      governing TinyCrypto's memory model, including the elimination of all
      heap-allocated state and reliance on compile-time-bounded stack frames.

  (2) A technical exposition of the cryptographic algorithm implementations,
      detailing constant-time coding disciplines, register-level memory management
      strategies, and ISA-specific optimization opportunities.

  (3) A description of the CMake-based cross-compilation pipeline supporting
      ARM Cortex-M0+, ARM Cortex-M4, and Xtensa LX6 target architectures.

  (4) A proposed tripartite benchmarking methodology for empirically evaluating
      TinyCrypto's performance across an x86 software baseline, a QEMU
      hardware-emulated context, and physical silicon targets.

The remainder of this paper is organized as follows. Section II surveys related
work and positions TinyCrypto within the existing landscape. Section III details
the system architecture and governing design constraints. Section IV presents
the implementation details. Section V describes the proposed benchmarking
methodology. Section VI concludes the paper.

================================================================================
II. RELATED WORK
================================================================================

A. OpenSSL
----------
OpenSSL is the de facto standard for cryptographic operations in general-purpose
computing. Its TLS stack provides a comprehensive implementation of NIST-approved
cipher suites, including AES-GCM, ChaCha20-Poly1305, RSA, and Elliptic Curve
Diffie-Hellman (ECDH). However, OpenSSL's architecture is intrinsically coupled
to a POSIX-compliant OS. At initialization, the OpenSSL library invokes OS
system calls for entropy acquisition (e.g., /dev/urandom or the getrandom(2)
syscall on Linux), dynamic memory management via malloc/free, file I/O for
configuration and certificate loading, and socket abstractions for TLS handshaking.
None of these primitives are available in a bare-metal firmware context. Attempts
to port OpenSSL to bare metal via stub implementations of the requisite system
calls have proven unreliable and impose a binary footprint that routinely exceeds
500 KB, manifestly excessive for a device with 256 KB of flash [6].

B. mbedTLS (Mbed TLS)
---------------------
Mbed TLS, developed by Arm, represents a significant advancement in embedded
cryptography. It is specifically designed for constrained environments and
supports a configurable build system that allows selective exclusion of unused
components. Mbed TLS can be configured to operate with a custom memory allocator,
partially mitigating the heap fragmentation issue. However, its primary
configuration and official documentation assume the presence of an RTOS for
thread-safe context management, and its default entropy source relies on the
mbedtls_hardware_poll() callback, which must be provided by a platform-specific
hardware abstraction layer (HAL). In comprehensive embedded configurations,
Mbed TLS still demands between 50 KB and 100 KB of flash for a functional TLS
stack, and its context structures (e.g., mbedtls_ssl_context, mbedtls_x509_crt)
are typically heap-allocated, making long-term bare-metal deployment fragile [7].

C. wolfSSL
----------
wolfSSL (formerly CyaSSL) is purpose-built for embedded platforms and achieves
an exceptionally compact footprint, with a TLS stack as small as 20-100 KB
of flash depending on the configuration. wolfSSL provides a FIPS 140-2 validated
cryptographic module and supports RTOS integrations including FreeRTOS, ThreadX,
and Nucleus. It also provides a NO_FILESYSTEM build mode, reducing POSIX
dependencies. Despite these optimizations, wolfSSL retains dynamic heap allocation
by default for session management, certificate chains, and TLS record processing,
and its RTOS-oriented design still imposes scheduling and context management
overhead that is non-trivial on cortex-M0 class processors [8].

D. Positioning TinyCrypto
--------------------------
Table I summarizes the comparison of TinyCrypto against the aforementioned
libraries across key embedded suitability metrics.

  TABLE I: Comparative Analysis of Embedded Cryptographic Libraries

  +------------------+----------+---------+---------+------------+
  | Feature          | OpenSSL  | mbedTLS | wolfSSL | TinyCrypto |
  +------------------+----------+---------+---------+------------+
  | OS Dependency    | Required | Partial | Partial | None       |
  | RTOS Dependency  | Required | Often   | Often   | None       |
  | Dynamic Alloc    | Yes      | Yes     | Yes     | None       |
  | Min Flash (KB)   | >500     | ~50     | ~20     | <10        |
  | Min SRAM (KB)    | >2000    | ~50     | ~20     | <4         |
  | Bare-Metal Ready | No       | Partial | Partial | Yes        |
  | Heap Fragile     | Yes      | Yes     | Yes     | No         |
  | Constant-Time    | Partial  | Partial | Partial | Yes        |
  +------------------+----------+---------+---------+------------+

TinyCrypto occupies a unique position: it is the only solution in this
comparison that is unequivocally OS-free, RTOS-free, heap-free, and designed
from first principles for bare-metal operation. It does not attempt to replicate
a full TLS stack but instead provides the subset of cryptographic primitives
sufficient to construct application-level secure channels optimal for deeply
embedded use cases.

III. SYSTEM ARCHITECTURE AND DESIGN CONSTRAINTS
================================================================================

A. The Zero-Dependency Design Philosophy
-----------------------------------------
The cardinal architectural principle of TinyCrypto is zero external linkage
at any level of the software stack. The engine is implemented in C++17, but
deliberately avoids all Standard Template Library (STL) containers, iterators,
and algorithms that rely on dynamic memory allocation or exception handling.
Specifically, the following compiler features are explicitly disabled at the
CMake configuration level:

  - Runtime Type Information (RTTI): Disabled via -fno-rtti.
  - C++ Exception Handling: Disabled via -fno-exceptions.
  - STL Heap Allocators: All std::vector, std::string, std::map usages
    are prohibited by coding convention enforced through static analysis.
  - Standard Library Linkage: The engine is compiled with -nostdlib and
    -nodefaultlibs, preventing any linkage against libc or libstdc++.

The result is a self-contained binary that links only against compiler builtins
(e.g., __aeabi_memcpy on ARM), ensuring portability and a deterministic,
bounded binary footprint.

B. Static Memory Model and Heap Allocation Avoidance
-----------------------------------------------------
The most critical vulnerability of cryptographic libraries operating in
long-running embedded firmware is heap fragmentation. When a system repeatedly
allocates and deallocates variable-size blocks of memory from a heap of bounded
size using a first-fit or best-fit allocator, the allocator's free-list
degenerates into a set of interspersed free regions that individually cannot
satisfy large allocation requests even when the aggregate free space is
sufficient. Over time horizons of weeks or months, this invariably leads to
allocation failure and system crash.

TinyCrypto eradicates this failure mode by forbidding heap allocation entirely.
Every cryptographic context in the engine is stack-allocated and governed by
automatic storage duration. For example, the AES round key schedule — a
14-round, 240-byte array derived from the 256-bit master key — is declared as
a compile-time-bounded C-style array on the caller's stack frame, and zeroed
via explicit_bzero() upon context destruction to preclude key material leakage
through stack residue. Similarly, the SHA-256 working state — an 8-element
array of 32-bit words plus the 64-byte message schedule — resides entirely
within the calling function's stack frame.

This design choice imposes a concrete requirement on callers: the stack partition
of the invoking task or ISR must be provisioned adequately. TinyCrypto publishes
compile-time constexpr constants describing the maximum stack depth required
for each primitive, enabling deterministic stack sizing at link time.

C. Cryptographic Suite
-----------------------
TinyCrypto implements the following cryptographic primitives:

  C.1 AES-256 in Cipher Block Chaining (CBC) Mode
  -------------------------------------------------
  The AES-256-CBC implementation follows the NIST FIPS 197 specification [9].
  The core SubBytes, ShiftRows, MixColumns, and AddRoundKey transformations
  are implemented using precomputed 1 KB lookup tables (S-Box and inverse S-Box)
  to achieve sub-microsecond round computation on 32-bit data paths. The key
  expansion (KeySchedule) produces 15 round key sub-arrays of 16 bytes each.
  The CBC mode employs a 128-bit Initialization Vector (IV), which must be
  cryptographically random for each message and never reused under the same key.
  Padding is handled via PKCS#7. The total code size of the AES module is
  approximately 2.8 KB of ARM Thumb-2 instructions.

  C.2 ChaCha20 Stream Cipher
  --------------------------
  The ChaCha20 implementation conforms to the RFC 8439 specification [10].
  ChaCha20 is particularly well-suited for embedded deployment because it
  requires no lookup tables (eliminating cache-timing side-channel attack
  vectors inherent in table-driven AES implementations on processors without
  data cache isolation) and operates exclusively using 32-bit add, XOR, and
  bit-rotation operations — all of which are single-cycle on Cortex-M4 and
  two cycles on Cortex-M0+. The quarter-round function is implemented as an
  inlined function to allow the compiler to schedule register allocations
  across rounds without stack spilling. The full 20-round ChaCha20 block
  function produces 64 bytes of keystream per invocation, with a maximum
  message length of 2^32 blocks per (key, nonce) pair.

  C.3 SHA-256 Hash Function
  --------------------------
  The SHA-256 implementation follows NIST FIPS 180-4 [11]. The 64-entry
  round constant table (K) is stored in flash (const uint32_t) to avoid
  SRAM consumption. The message schedule W[t] is computed in-place using
  a circular 16-element buffer to reduce the working memory requirement
  from 256 bytes (64 * 4) to 64 bytes. The final hash digest is output
  as a 32-byte big-endian array. Total stack usage during hashing is
  bounded at approximately 192 bytes.

  C.4 Curve25519 Elliptic Curve Diffie-Hellman
  ---------------------------------------------
  Curve25519 [12] provides the asymmetric cryptographic foundation for
  future key exchange extensions. The implementation performs scalar
  multiplication on the Montgomery curve y² = x³ + 486662x² + x over
  the prime field GF(2^255 - 19) using the Montgomery ladder algorithm [13].
  The Montgomery ladder is chosen for its inherent resistance to simple
  power analysis (SPA) side-channel attacks, as it performs a fixed number
  of field operations regardless of the scalar bit pattern, ensuring constant-
  time execution. The field arithmetic is implemented using 256-bit integers
  represented as eight 32-bit limbs, with all reduction steps performed
  explicitly to avoid overflow on 32-bit ALUs without hardware multipliers.

D. Cross-Compilation Pipeline
------------------------------
The cross-compilation pipeline is orchestrated entirely via CMake (minimum
version 3.16) using a toolchain-file abstraction pattern. Separate toolchain
files are provided for each supported target ISA:

  - arm_cortex_m0plus.cmake: Configures arm-none-eabi-gcc with flags
    -mcpu=cortex-m0plus -mthumb -mfloat-abi=soft. Enables LTO and
    dead-code elimination (-ffunction-sections -fdata-sections) with
    --gc-sections at link time. Disables the C/C++ startup runtime
    (-nostartfiles), replacing the default __weak reset handler with a
    minimal crt0 that initializes .bss to zero and copies .data from flash
    before invoking the application entry point.

  - xtensa_lx6_esp32.cmake: Configures the Xtensa toolchain
    (xtensa-esp32-elf-gcc) with -mlongcalls to resolve inter-section
    call-distance limitations inherent to the Xtensa ISA's fixed 18-bit
    call offset field. The ESP32's dual-core architecture is explicitly not
    leveraged; TinyCrypto operates on a single core to eliminate the need
    for synchronization primitives.

  - x86_64_host.cmake: Configures the native host compiler (GCC or Clang)
    for unit testing, benchmark baselining, and Valgrind-based memory
    analysis purposes.

The CMake superbuild pattern is employed to ensure that all targets share
identical source files, with ISA-specific optimizations gated via preprocessor
conditionals (TINYCRYPTO_ARCH_ARM_M0, TINYCRYPTO_ARCH_ARM_M4,
TINYCRYPTO_ARCH_XTENSA_LX6) injected by the respective toolchain files.

================================================================================
IV. IMPLEMENTATION DETAILS
================================================================================

A. Memory Register Management
-------------------------------
TinyCrypto's implementation disciplines are informed by the physical
characteristics of the target register files. The ARM Cortex-M0+ provides 16
general-purpose 32-bit registers (R0–R15), of which 13 (R0–R12) are freely
usable by the compiler's register allocator; R13 is the stack pointer, R14 the
link register, and R15 the program counter. The M0+ critically lacks the
hardware division instruction and SIMD (Single Instruction, Multiple Data)
extensions present in Cortex-M4. TinyCrypto's ChaCha20 implementation is
therefore written to minimize register pressure, keeping the four ChaCha quarter-
round state variables (a, b, c, d) in R0–R3 across the 20-round loop to avoid
spilling working state to the stack, which would incur two additional memory
bus cycles per spill/reload pair.

For the AES-256 implementation on Cortex-M4 targets, the architecture's
barrel shifter is exploited to implement 8-bit S-Box lookups via 32-bit aligned
table accesses, reducing the round transformation loop from 16 byte-granularity
lookups to 4 word-granularity operations per row. This reduces the instruction
count by a factor of approximately 4x compared to the byte-granularity
implementation required on M0+ targets.

All sensitive key material — round key schedules, ephemeral ChaCha20 state
words, and SHA-256 working variables — is explicitly memset to zero immediately
prior to function return using the volatile qualifier or a platform-specific
explicit_bzero() equivalent to preclude dead-store elimination by sufficiently
aggressive compiler optimizers. This practice is mandated by the coding
standard enforced across the entire TinyCrypto codebase and constitutes a
defense-in-depth measure against key material recovery via stack probing
or cold-boot attacks.

B. Constant-Time Execution Discipline
---------------------------------------
All cryptographic operations in TinyCrypto are implemented to execute in time
independent of secret data values, thereby mitigating timing side-channel attacks.
Specifically:

  (1) Secret-dependent branching: All conditional branches dependent on key
      material, plaintext, or intermediate cipher state are replaced with
      branchless, bitwise equivalents. For example, conditional byte selection
      is implemented as: result = (condition_mask & value_a) | (~condition_mask
      & value_b), where condition_mask is derived via arithmetic rather than
      comparison.

  (2) Table-driven AES and cache-timing attacks: On platforms without data
      cache isolation, the table-based AES implementation leaks key-dependent
      access patterns observable by cache-timing adversaries. TinyCrypto
      provides a compile-time option (TINYCRYPTO_AES_BITSLICED=1) to substitute
      a bitsliced AES implementation that operates entirely within the register
      file, eliminating all memory-dependent timing variation at the cost of
      approximately 2x the instruction count.

  (3) Montgomery ladder for Curve25519: As detailed in Section III-C.4, the
      Montgomery ladder ensures that scalar multiplication in Curve25519 performs
      a fixed sequence of field operations for any scalar value, providing
      resistance against simple power analysis.

C. Pre-Silicon Validation via QEMU Hardware Emulation
------------------------------------------------------
Prior to deployment on physical silicon, TinyCrypto firmware images can be
validated using QEMU, the open-source machine emulator and virtualizer [14].
QEMU provides a faithful instruction-level simulation of ARM Cortex-M series
processors, including cycle-accurate emulation of the ARMv6-M instruction set
(used by Cortex-M0+) and ARMv7-M (used by Cortex-M4). The emulation supports
semihosting, a mechanism by which debugger-intercepted halt instructions (BKPT
#0xAB on ARM) relay I/O operations from the emulated firmware to the host OS,
enabling printf-style diagnostic output without a physical UART.

The QEMU-based test harness within TinyCrypto operates as follows:

  (1) The cross-compiled ELF binary (e.g., libtinycrypto_test.elf, targeted
      for the RP2040's ARM Cortex-M0+ ISA) is loaded into QEMU using the
      qemu-system-arm -machine microbit (or equivalent cortex-m board model)
      command.

  (2) Known Answer Tests (KATs) derived from NIST Cryptographic Algorithm
      Validation Program (CAVP) test vectors [15] are executed. The results
      are transmitted via the semihosting interface and compared against
      expected values by a host-side Python validation script.

  (3) The QEMU GDB server mode (qemu-system-arm -gdb tcp::1234 -S) is used
      in conjunction with arm-none-eabi-gdb to set watchpoints on sensitive
      memory regions, verifying that key material is correctly zeroed upon
      context destruction before returning to the caller.

  (4) Once all KATs pass in QEMU, the firmware is flashed to physical silicon
      using the appropriate programming utility (e.g., OpenOCD for STM32,
      picotool for RP2040) and regression tests are re-run over a hardware
      UART interface.

This emulation-first development workflow significantly accelerates the
embedded development cycle, eliminating the costly flash-halt-debug iteration
loop on physical hardware during initial algorithm validation.

V. PROPOSED BENCHMARKING METHODOLOGY
================================================================================

A. Evaluation Objectives
--------------------------
The benchmarking campaign serves three primary objectives:

  (1) Quantify the absolute execution latency (in processor clock cycles and
      wall-clock milliseconds) for each cryptographic primitive across all
      three evaluation environments.

  (2) Measure peak SRAM consumption for each primitive's invocation, using
      stack watermarking to determine the highest stack address written during
      execution.

  (3) Compare code-size (text segment) footprint in kilobytes across ISA targets
      to quantify the flash memory budget required for TinyCrypto integration.

B. Evaluation Environments
---------------------------
The proposed benchmarking evaluates TinyCrypto across three distinct hardware
and software execution contexts:

  B.1 Standard x86-64 Host CPU (Software Baseline)
  --------------------------------------------------
  The host-compiled (x86-64) binary, assembled with GCC 12 at -O2 optimization,
  is treated as a pure computational baseline. Performance is measured using
  the POSIX clock_gettime(CLOCK_MONOTONIC) API with nanosecond resolution.
  Each primitive is exercised over 10,000 iterations; the arithmetic mean and
  standard deviation of execution time are reported. This environment does not
  reflect embedded deployment constraints but establishes a scalable performance
  anchor against which relative ISA overhead can be quantified.

  B.2 QEMU Hardware Emulator (Simulated Embedded Target)
  -------------------------------------------------------
  The cross-compiled ARM Cortex-M0+ binary is executed under QEMU
  (qemu-system-arm, version 8.x). QEMU's instruction counting plugin
  (-plugin libqemu-plugin-insn-count.so) enumerates the precise number of
  architectural instructions retired per cryptographic operation, providing
  an ISA-normalized performance metric independent of host clock speed.
  Memory usage is assessed by instrumenting the firmware to periodically
  sample the current stack pointer value and record the minimum observed
  address, from which peak stack depth is derived.

  B.3 Physical Silicon (Raspberry Pi Pico, RP2040)
  -------------------------------------------------
  The Raspberry Pi Pico, based on the RP2040 SoC (dual ARM Cortex-M0+ at
  133 MHz, 264 KB SRAM), serves as the primary physical silicon target. The
  firmware is flashed via picotool over USB in BOOTSEL mode. Performance
  measurements are obtained using the RP2040's internal 64-bit hardware timer
  (timer_hw->timerawl / timer_hw->timerawh) read before and after each
  primitive invocation, providing microsecond-resolution timestamps directly
  from hardware registers. Stack consumption is measured using a stack canary
  pattern: the allocatable stack region is pre-filled with the sentinel value
  0xDEADC0DE, and after the cryptographic operation completes, the unmodified
  canary region is scanned from the stack base upward to determine the high-
  water mark of actual stack usage.

C. Reported Metrics
--------------------
For each (algorithm, environment) combination, the following metrics will be
tabulated:

  TABLE II: Proposed Benchmark Metrics (per algorithm, per environment)

  +-------------------------------+--------------------------------------------+
  | Metric                        | Description                                |
  +-------------------------------+--------------------------------------------+
  | Execution Time (us)           | Wall-clock time per single invocation      |
  | Throughput (KB/s)             | Payload processed per second               |
  | Clock cycles                  | Processor cycles per byte (Cpb)            |
  | Peak Stack Usage (bytes)      | Maximum bytes of stack consumed            |
  | Code Size (bytes)             | Text segment size of algorithm module      |
  | Const-time Verified           | Boolean: no secret-dependent branches      |
  +-------------------------------+--------------------------------------------+

D. Benchmark Test Suite
------------------------
The benchmark suite evaluates the following operations:

  - AES-256-CBC: 16-byte, 256-byte, and 1024-byte encryption and decryption.
  - ChaCha20: 64-byte (one block) and 1024-byte keystream generation.
  - SHA-256: Empty message, 55-byte (single-block), and 1024-byte hashing.
  - Curve25519: Single scalar multiplication on the base point.

A secondary benchmark evaluates a compound "secure message" scenario combining
ChaCha20 encryption followed by SHA-256 authentication tag computation,
representative of an application-layer authenticated encryption workflow.

================================================================================
VI. CONCLUSION
================================================================================

The security of the global IoT ecosystem cannot be guaranteed through the
uncritical application of desktop cryptographic libraries to hardware platforms
with categorically incompatible computational constraints. TinyCrypto represents
a principled engineering response to this challenge: a zero-dependency, bare-
metal C++ cryptographic engine that operates directly on physical silicon without
an OS, RTOS, or heap allocator, delivering a suite of standardized cryptographic
primitives — AES-256-CBC, ChaCha20, SHA-256, and Curve25519 — at the extreme
edge of the network.

By enforcing a static memory model, suppressing the compiler's Standard Library
linkage, and implementing constant-time algorithm disciplines, TinyCrypto
eliminates the three most critical failure modes of embedded cryptographic
deployments: heap fragmentation-induced crashes, OS-dependency incompatibility,
and timing side-channel key recovery. The CMake-driven cross-compilation pipeline
ensures that a single, audited codebase can be deterministically compiled and
deployed across ARM Cortex-M0+, ARM Cortex-M4, and Xtensa LX6 targets, with
pre-silicon validation through QEMU emulation reducing integration risk.

The proposed tripartite benchmarking methodology — spanning x86 software
baseline, QEMU emulation, and the Raspberry Pi Pico physical silicon target —
will provide the embedded security community with a rigorous, reproducible
empirical characterization of TinyCrypto's performance envelope relevant to
real-world deployment on constrained microcontrollers.

As the embedded attack surface continues to expand alongside the proliferation
of autonomously operating IoT nodes, TinyCrypto establishes a foundational
security primitive that can serve as the basis for application-layer secure
channels, attestation protocols, and authenticated firmware update mechanisms
on the most resource-constrained hardware platforms in deployment today.
